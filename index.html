03. 역할 및 기여도

팀장 (기획 총괄 및 백엔드/AI 리드)

  • 100% AI 챗봇 엔진과 MongoDB 기반 스키마 설계 및 백엔드 로직 전담
  • 80% 부동산 및 인프라 원천 데이터의 수집, 전처리, DB 적재 파이프라인 구축
  • 70% 서비스에 최적화된 다중 조건 가중합 스코어링 알고리즘 발굴 및 추천 로직 적용

04. 사용 기술 및 도입 이유

Python

풍부한 데이터 처리 생태계를 바탕으로 웹 크롤링부터 AI 로직까지 통합 구현하기 위해 채택

Flask

가볍고 유연한 마이크로 프레임워크로, Python 기반의 AI 모듈 및 데이터 스크립트와 가장 빠르게 연동하기 위해 사용

MongoDB Atlas

NoSQL 특유의 유연한 스키마로 데이터를 직관적으로 관리하며, 2dsphere 공간 인덱스를 활용해 위치 기반 검색의 처리 속도를 극대화

Web Crawling (Selenium)

3주차: AI & 백엔드 구현 LangChain과 GPT를 연동하여, 사용자의 자연어를 MongoDB 검색용 JSON 필터로 자동 변환하는 챗봇 엔진 및 검색 API 구현

06. 결과 및 성과

추천 파이프라인 완성

단순 물리적 스펙(가격/위치) 필터링을 넘어, 사용자의 라이프스타일 설문과 주변 인프라 데이터를 결합한 '다중 조건 가중합 스코어링 모델'을 자체 설계하여 맞춤형 Top-5 매물 추천 로직을 성공적으로 구현했습니다.

자연어 기반 UX 혁신

LLM과 LangChain을 활용해 사용자의 일상적인 대화(자연어)를 MongoDB 검색용 JSON 필터로 자동 파싱하는 챗봇 엔진을 개발하여, 기존의 복잡한 다단계 UI 필터 조작 과정을 텍스트 입력 하나로 단축시켰습니다.

DB 성능 80% 향상

수만 건의 매물 및 상권 데이터 검색 시 발생하는 병목 현상을 해결하기 위해 공간 인덱싱(2dsphere)을 적용하여, 평균 데이터 조회 시간을 0.2초 이내로 단축하는 기술적 성과를 달성했습니다.

07. 트러블슈팅 (어려움 및 해결 과정)

발생한 문제 및 원인

[문제점] AI 챗봇 구현 초기, 사용자가 이전에 입력한 매물 필터링 조건이 대화 도중 초기화되거나, 실제 DB에 존재하지 않는 가상의 지명을 추천하는 오류가 발생했습니다.

[원인] LLM 특유의 상태 비저장(Stateless) 구조로 인한 맥락 휘발성 문제와, 허위 정보를 생성하는 할루시네이션(환각) 현상 때문이었습니다.

해결 과정
해결 1.

상태 누적 기반 맥락 유지: 대화의 흐름이 끊기지 않도록 '상태 누적(State Accumulation) 방식'의 변수를 도입했습니다. 사용자의 이전 필터링 조건과 대화 맥락을 시스템에 지속적으로 유지하여 조건 초기화 문제를 해결했습니다.

해결 2.

프롬프트 제약을 통한 환각 차단: 시스템 프롬프트에 '반드시 DB 검색을 통해 도출된 결과 데이터 내에서만 답변을 생성할 것'이라는 강력한 제약 조건을 부여하여, AI가 임의의 지명을 지어내는 할루시네이션 문제를 완벽하게 통제했습니다.

08. 배운 점

LLM(대형 언어 모델)은 만능이 아니며, 이를 실제 상용 서비스 레벨로 끌어올리기 위해서는 모델의 태생적 한계(맥락 휘발성, 환각 현상)를 명확히 이해해야 함을 깨달았습니다. 단순히 API를 호출하는 것을 넘어, 상태 관리(State Management) 로직과 정교한 프롬프트 엔지니어링을 통해 AI를 완벽하게 통제하는 '엔지니어링 역량'의 중요성을 실무적으로 체득했습니다.

Gemini API

React & Tailwind CSS

부동산 추천 AI 컨시어지 서비스

팀 프로젝트

사용자의 자연어 대화와 라이프스타일을 분석하여
최적의 부동산 매물을 찾아주는 AI 컨시어지 서비스

02. 배경 및 목표

기존 부동산 서비스의 물리적 스펙 위주 검색이 가진 한계를 벗어나고자 했습니다. 인프라를 중시하는 2030 1인 가구를 타겟으로, 간단한 설문을 통해 선호하는 라이프스타일을 파악하고 이에 꼭 맞는 매물을 추천하는 맞춤형 주거 큐레이션 서비스를 목표로 했습니다.

03. 역할 및 기여도

팀장 (기획 총괄 및 백엔드/AI 리드)

  • 100% AI 챗봇 엔진과 MongoDB 기반 스키마 설계 및 백엔드 로직 전담
  • 80% 부동산 및 인프라 원천 데이터의 수집, 전처리, DB 적재 파이프라인 구축
  • 70% 서비스에 최적화된 다중 조건 가중합 스코어링 알고리즘 발굴 및 추천 로직 적용

04. 사용 기술 및 도입 이유

Python

풍부한 데이터 처리 생태계를 바탕으로 웹 크롤링부터 AI 로직까지 통합 구현하기 위해 채택

Flask

가볍고 유연한 마이크로 프레임워크로, Python 기반의 AI 모듈 및 데이터 스크립트와 가장 빠르게 연동하기 위해 사용

MongoDB Atlas

NoSQL 특유의 유연한 스키마로 데이터를 직관적으로 관리하며, 2dsphere 공간 인덱스를 활용해 위치 기반 검색의 처리 속도를 극대화

LangChain

복잡한 프롬프트 체이닝을 모듈화하고, 사용자의 자연어를 검색 쿼리로 변환하는 LLM 파이프라인을 효율적으로 구축하기 위해 도입

OpenAI API (GPT-4o-mini)

실시간 챗봇 서비스에 필수적인 '빠른 연산 속도'와 대규모 텍스트 처리에 적합한 '저렴한 토큰 비용'을 모두 충족

Web Crawling (Selenium)

자바스크립트(JS)로 렌더링되는 동적 웹 페이지의 부동산 데이터를 안정적으로 수집하기 위해 활용

05. 진행 과정

1주차: 기획 및 아키텍처 설계 타겟 사용자 분석을 통한 서비스 요구사항 정의 및 '다중 조건 가중합 스코어링' 추천 로직 구상
2주차: 데이터 파이프라인 구축 크롤링 및 공공데이터를 결합해 약 7만 건(매물 3만, 인프라 4만)의 원천 데이터 수집·전처리 후, MongoDB 스키마 설계 및 데이터 적재
3주차: AI & 백엔드 구현 LangChain과 GPT를 연동하여, 사용자의 자연어를 MongoDB 검색용 JSON 필터로 자동 변환하는 챗봇 엔진 및 검색 API 구현
4주차: QA 및 데이터 튜닝 통합 테스트를 통한 버그 수정 진행. 추천 정확도 향상을 위한 스코어링 가중치 파라미터의 세밀한 튜닝 및 검색 성능 최적화 반복
협업 및 형상 관리 4인 프로젝트 팀장으로서 GitHub Branch 전략을 도입해 충돌 없이 코드를 관리하고, Notion과 Slack을 활용해 일정과 이슈를 애자일하게 공유하며 소통함

06. 결과 및 성과

추천 파이프라인 완성

단순 물리적 스펙(가격/위치) 필터링을 넘어, 사용자의 라이프스타일 설문과 주변 인프라 데이터를 결합한 '다중 조건 가중합 스코어링 모델'을 자체 설계하여 맞춤형 Top-5 매물 추천 로직을 성공적으로 구현했습니다.

자연어 기반 UX 혁신

LLM과 LangChain을 활용해 사용자의 일상적인 대화(자연어)를 MongoDB 검색용 JSON 필터로 자동 파싱하는 챗봇 엔진을 개발하여, 기존의 복잡한 다단계 UI 필터 조작 과정을 텍스트 입력 하나로 단축시켰습니다.

DB 성능 80% 향상

수만 건의 매물 및 상권 데이터 검색 시 발생하는 병목 현상을 해결하기 위해 공간 인덱싱(2dsphere)을 적용하여, 평균 데이터 조회 시간을 0.2초 이내로 단축하는 기술적 성과를 달성했습니다.

07. 트러블슈팅 (어려움 및 해결 과정)

발생한 문제 및 원인

[문제점] AI 챗봇 구현 초기, 사용자가 이전에 입력한 매물 필터링 조건이 대화 도중 초기화되거나, 실제 DB에 존재하지 않는 가상의 지명을 추천하는 오류가 발생했습니다.

[원인] LLM 특유의 상태 비저장(Stateless) 구조로 인한 맥락 휘발성 문제와, 허위 정보를 생성하는 할루시네이션(환각) 현상 때문이었습니다.

해결 과정
해결 1.

상태 누적 기반 맥락 유지: 대화의 흐름이 끊기지 않도록 '상태 누적(State Accumulation) 방식'의 변수를 도입했습니다. 사용자의 이전 필터링 조건과 대화 맥락을 시스템에 지속적으로 유지하여 조건 초기화 문제를 해결했습니다.

해결 2.

프롬프트 제약을 통한 환각 차단: 시스템 프롬프트에 '반드시 DB 검색을 통해 도출된 결과 데이터 내에서만 답변을 생성할 것'이라는 강력한 제약 조건을 부여하여, AI가 임의의 지명을 지어내는 할루시네이션 문제를 완벽하게 통제했습니다.

08. 배운 점

LLM(대형 언어 모델)은 만능이 아니며, 이를 실제 상용 서비스 레벨로 끌어올리기 위해서는 모델의 태생적 한계(맥락 휘발성, 환각 현상)를 명확히 이해해야 함을 깨달았습니다. 단순히 API를 호출하는 것을 넘어, 상태 관리(State Management) 로직과 정교한 프롬프트 엔지니어링을 통해 AI를 완벽하게 통제하는 '엔지니어링 역량'의 중요성을 실무적으로 체득했습니다.
